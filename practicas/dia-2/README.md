# Prácticas — Día 2

## Objetivo del día

Practicar el flujo de trabajo colaborativo en GitHub mediante ramas y Pull Requests.

## Actividades realizadas

- Creación de una rama nueva llamada `docs-practica-dia-2`.
- Trabajo separado de la rama principal `main`.
- Creación de documentación diaria en Markdown.
- Preparación de un Pull Request para proponer cambios.
- Revisión del concepto de rama, commit, Pull Request y merge.

## Conceptos trabajados

| Concepto | Explicación |
|---|---|
| Rama | Línea de trabajo separada de `main` |
| Pull Request | Propuesta de cambio para revisar antes de integrar |
| Compare | Comparación entre una rama y otra |
| Merge | Unión de los cambios de una rama dentro de `main` |
| Revisor | Persona que revisa los cambios antes de aprobarlos |

## Flujo aplicado

1. Partir de la rama `main`.
2. Crear una rama nueva para la práctica.
3. Añadir documentación en la rama nueva.
4. Hacer commit con un mensaje descriptivo.
5. Abrir un Pull Request.
6. Revisar los cambios.
7. Fusionar la rama con `main`.

## Evidencias

- Rama `docs-practica-dia-2` creada correctamente.
- Archivo `practicas/dia-2/README.md` creado.
- Commit realizado en una rama diferente de `main`.
- Pull Request preparado para revisión.

## Reflexión

En esta práctica he aprendido que trabajar con ramas permite realizar cambios sin modificar directamente la versión principal del repositorio.

El Pull Request sirve como mecanismo de revisión antes de integrar cambios en `main`, lo que ayuda a trabajar de forma más ordenada, segura y profesional.
